# CM-012 Gap Priorities

Clasificación de gaps CM-011 para piloto V2. No se aplicaron cambios remotos.

| ID | Prioridad | Descripción | Módulo | Impacto | Solución recomendada | Riesgo | Requiere migración | Requiere infraestructura | Requiere secreto | Requiere costo | Requiere aprobación humana |
|---|---:|---|---|---|---|---|---|---|---|---|---|
| GAP-01 | P0 | Upload ciudadano de evidencia sin Edge Function/contrato seguro | Citizen Portal / Storage | Impide evidencia ciudadana real segura; flujo puede operar sin evidencia inicial | Crear Edge Function validando MIME, tamaño, tenant, path `institution_id/pending/...` y asociación posterior | Abuso de Storage, costos, bypass de validación | YES | YES | YES | Posible | YES |
| GAP-02 | P0 | Buckets privados `ticket-evidence-v11` y `resolution-evidence-v11` son prerrequisito manual | Storage & Evidence | Bloquea evidencia real si no existen en Supabase aprobado | Crear buckets privados y aplicar policies en ambiente autorizado siguiendo checklist | Upload/lectura fallarán aunque RPC exista | NO SQL necesariamente | YES | NO | Posible storage | YES |
| GAP-03 | P1 | Falta RPC específica para resolver brigada activa del usuario | Brigade Portal | Portal puede funcionar con consultas existentes, pero la UX/contexto de brigada es frágil | Añadir RPC `v11_get_my_brigade_context` o vista segura por RLS | Asignación equivocada o ambigua en usuarios multi-brigada | YES | NO | NO | NO | YES |
| GAP-04 | P1 | Administración transaccional incompleta | Master Admin | Alta/modificación de instituciones, memberships, brigadas y usuarios no está lista para operación segura desde UI | Diseñar RPCs/Edge Functions administrativas con auditoría | Cambios directos pueden ser inseguros o incompletos | YES | Posible | Posible | NO | YES |
| GAP-05 | P1 | No hay usuarios, memberships, brigadas y datos piloto autorizados | Auth / Municipal / Brigade | Impide pruebas reales E2E | Crear paquete de datos piloto aprobado o runbook manual | Datos reales mal configurados rompen RLS | Posible | NO | NO | NO | YES |
| GAP-06 | P1 | No hay configuración pública Supabase autorizada en repo/entorno | Todos | Pruebas reales quedan REAL_NOT_RUN | Definir variables públicas `SAIBOT_SUPABASE_URL` y anon key por entorno | Conectar entorno equivocado o exponer valores no aprobados | NO | YES | NO (anon pública) | NO | YES |
| GAP-07 | P2 | Notificaciones reales no tienen proveedor, worker, secrets ni retries | Notifications | No bloquea flujo ticket principal; sí afecta piloto operativo | Mantener simulador hasta aprobar proveedor y worker | Envíos accidentales/costos si se improvisa | Posible | YES | YES | YES | YES |
| GAP-08 | P2 | Realtime no confirmado | Todos | UX no se actualiza en vivo; puede usar refresh manual | Diseñar canales por institución y validar RLS | Riesgo cross-tenant si se improvisa | Posible | NO | NO | NO | YES |
| GAP-09 | P2 | Escritura de configuración desde UI requiere revisión | Configuration | Piloto puede preconfigurarse manualmente; autogestión limitada | RPCs de configuración con auditoría y validación | Config pública inconsistente | YES | NO | NO | NO | YES |
| GAP-10 | P3 | Correlation IDs explícitos no están formalizados | Audit / Observabilidad | No bloquea piloto; limita trazabilidad avanzada | Agregar columna/propagación de correlation id | Diagnóstico menos preciso | YES | NO | NO | NO | YES |
